# Git 多地协作最佳实践指南
——告别“文件丢失”与“版本覆盖”

为了实现“在公司写一部分，回家接着写，再回公司无缝衔接”的理想工作流，请务必遵守以下操作规范。

---

## 🕒 场景一：上班/开机第一件事（拉取）

在你打开电脑，准备开始工作**之前**，请务必执行：

```bash
git pull origin main
```

*   **作用**：把你在另一个地方（比如家里）写的最新代码拉取到当前电脑。
*   **如果不做会怎样？** 你可能会基于旧版本修改代码，导致后续推送时产生冲突，甚至覆盖掉你在另一处的修改。

---

## 🕒 场景二：下班/关机最后一件事（推送）

在你结束工作，准备关闭电脑**之前**，请务必执行：

```bash
git add .
git commit -m "今日工作进度备份"
git push origin main
```

*   **作用**：把当前电脑上的最新修改同步到云端（GitHub）。
*   **注意**：`commit -m` 后面的引号里写什么都可以，建议写清楚进度，比如“完成数据清洗”、“修复TFP计算Bug”。

---

## 🚨 场景三：如果忘记拉取就修改了怎么办？

如果你在公司写了一天代码，突然想起：“糟糕！昨晚在家写的代码还没拉下来！”，这时候直接 `push` 会报错。

**补救措施**：
1.  **先尝试拉取并合并**：
    ```bash
    git pull origin main --rebase
    ```
    *   如果顺利，Git 会自动把你的新修改“嫁接”到远程代码之后。
    *   如果有冲突（Conflict），IDE 会提示你解决冲突（保留哪部分代码）。

2.  **强制覆盖（慎用！）**：
    *   如果你确定**不需要**另一处的代码了（比如家里的代码写废了），可以使用：
    *   `git push origin main -f`
    *   **警告**：这会永久删除远程仓库里那些你本地没有的提交。

---

## 📝 总结口诀

> **开机先 Pull，关机再 Push。**
> **多地办公，勤提交，不冲突。**

只要养成这个习惯，你的代码就会像云盘一样，随时随地都是最新的！🚀
